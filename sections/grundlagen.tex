\section{Grundlagen}
\label{sec:grundlagen}

\subsection{Multi-Tenant SaaS-Architektur}

dnchub-app ist so gebaut, dass sich mehrere Organisationen eine einzige
laufende Anwendungsinstanz samt einer gemeinsamen Datenbank teilen. Jede
Zeile in einer geschäftsrelevanten Tabelle trägt ein zusätzliches Feld
namens \texttt{organization\_id}, das als Unterscheidungsmerkmal dient.
Damit Mandant A niemals Daten von Mandant B zu sehen bekommt, muss jede
SQL-Abfrage dieses Feld in ihre \texttt{WHERE}-Klausel aufnehmen und es
mit dem Wert aus dem Session-Token des eingeloggten Benutzers abgleichen.
Wird dieser Abgleich zentral an einer Stelle vergessen, die später von
vielen Endpunkten verwendet wird, vervielfacht sich die Lücke praktisch
auf das gesamte Modul.

\subsection{Insecure Direct Object Reference (IDOR)}

IDOR bezeichnet eine Zugriffskontrolllücke, bei der eine Anwendung
einen vom Client frei wählbaren Bezeichner, etwa eine UUID im URL-Pfad,
ohne Berechtigungsabgleich direkt an die Datenbank weiterreicht
\parencite{owasp_top10_a01}. Der Aufrufer kann den Bezeichner
austauschen und gelangt so an Daten, die ihm nicht gehören. Wie sich
das in dnchub-app konkret äussert, zeigt der folgende
Ablauf in fünf Schritten:

\begin{enumerate}
      \item Angreifer authentifiziert sich als Benutzer von Organisation A.
      \item Über einen normalen API-Aufruf erhält er eine gültige UUID eines
            eigenen Ressourceneintrags (z.\,B. eines Fahrzeugs).
      \item Er ersetzt in einem weiteren Request die UUID durch eine fremde UUID
            (erraten, durch Enumeration oder durch Insiderwissen erhalten).
      \item \texttt{BaseService.get\_or\_404(db, uuid)} fragt die Datenbank mit
            \texttt{WHERE id = uuid} ab, ohne \texttt{organization\_id}-Prüfung.
      \item Die Response enthält die Ressource von Organisation B.
\end{enumerate}

Da alle PATCH- und DELETE-Endpunkte ebenfalls über \texttt{get\_or\_404()}
gehen, kann der Angreifer fremde Datensätze auch verändern oder löschen.

\subsection{Soft-Delete-Muster}

Wenn ein Benutzer in dnchub-app auf \enquote{Löschen} klickt, verschwindet
der betroffene Eintrag nicht wirklich aus der Datenbank. Stattdessen wird
die Spalte \texttt{deleted\_at} mit dem aktuellen Zeitstempel befüllt und
die Zeile bleibt physisch erhalten. Dieses Vorgehen hat zwei praktische
Vorteile. Zum einen lassen sich so Audit-Trails erstellen, zum anderen
kann ein versehentlich gelöschter Datensatz ohne Backup wiederhergestellt
werden. Der Ansatz funktioniert allerdings nur unter einer Bedingung. Jede
lesende Abfrage muss die Klausel \texttt{deleted\_at IS NULL} enthalten,
sonst verhält sich die \enquote{Löschung} gegenüber dem Aufrufer wie eine
blosse Markierung. Genau diese Klausel fehlt in den Methoden
\texttt{get()} und \texttt{get\_or\_404()}, was Issue~\#14
\parencite{github_issue_14} als eigenständige Schwachstelle einordnet.
